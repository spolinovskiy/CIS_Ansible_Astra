# Astra Linux OS Mapping

Generated by PRD task `ASTRA-0005` at `2026-04-27T22:08:37Z`.

## Scope

These files define Astra Linux detection, package, service, path, command, and feature mappings for later audit tasks. They do not implement CIS controls and do not add scope beyond `data/cis_linux_scope/controls_from_excel.json`.

Reference-only interpretation may use the CIS Debian 12 and Ubuntu 20.04 benchmarks, but the authoritative scope remains the extracted Excel control list.

## Files

- `data/os_map/supported_versions.yml`: Astra fact detection and candidate version handling.
- `data/os_map/packages.yml`: apt/dpkg-oriented package map, marked unverified for Astra until lab evidence exists.
- `data/os_map/services.yml`: systemd service unit candidates for SSH, cron/at, time sync, logging, audit, and related facilities.
- `data/os_map/paths.yml`: configuration and evidence paths for sudo, PAM, SSH, cron/at, time sync, journald, rsyslog, auditd, bootloader, mounts, sysctl, and kernel modules.
- `data/os_map/commands.yml`: read-only audit command candidates and guarded remediation-only command references.
- `data/os_map/features.yml`: high-level capability map consumed by future chapter implementations.

## Version Handling

Future playbooks must use these Ansible facts before selecting Astra-specific behavior:

- `ansible_distribution`
- `ansible_distribution_version`
- `ansible_distribution_major_version`
- `ansible_os_family`
- `ansible_kernel`

The map accepts Astra only when `ansible_distribution` identifies Astra Linux. Common Astra version families `1.6`, `1.7`, `1.8`, and `2.x` are recorded as `candidate_untested`; no version is marked as tested or production-supported by this task.

Unknown Astra versions may be audited read-only if the host is clearly Astra Linux. Remediation and package operations must remain blocked until the exact version is approved in required information.

## Safety Guards

The following guard variables are mandatory for future remediation logic:

- `remediation_enabled`: defaults to `false` and is required for any change.
- `package_install_allowed`: defaults to `false` and is required for package install, update, or removal.
- `local_repository_defined`: defaults to `false` and is required before any apt/dpkg package operation that needs repository metadata.

Service restarts, SSH changes, bootloader updates, PAM changes, audit/logging changes, and sysctl persistence changes require additional engineer approval and rollback information from `data/required_information/required_information_template.yml`.

## Caveats

- Astra package names, service unit names, and Debian-like paths are candidates unless explicitly marked otherwise.
- apt/dpkg and systemd are treated as likely for Astra, not verified in this repository.
- Bootloader paths vary by BIOS/UEFI and must be detected per host.
- PAM paths are Debian-like candidates and must not be modified without Astra lab validation.
- No public repository URL, package source, service baseline, production value, or tested version was invented.
